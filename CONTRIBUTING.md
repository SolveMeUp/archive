# 기여 가이드

`archive`는 [SolveMeUp](https://solvemeup.com)의 자료구조·알고리즘 지식 아카이브입니다.
누구나 글을 제안할 수 있고, 큐레이터의 리뷰를 거쳐 반영됩니다.

## 라이선스 동의 (중요)

이 저장소에 기여하면, 본인의 기여물이 저장소와 동일한 조건으로 제공되는 데 동의하는 것으로 간주합니다.
별도의 CLA는 없습니다.

- **글(산문)** 은 [CC BY-SA 4.0](LICENSE)으로 제공됩니다.
- **코드 예제(스니펫)** 는 자유 사용으로 제공됩니다. (저작권을 주장하지 않습니다)

## 제안 방법

직접 push 권한은 큐레이터에게만 있습니다. 그 외에는 **fork → Pull Request** 흐름을 따릅니다.

1. 이 저장소를 fork 합니다.
2. 브랜치를 만들어 작업합니다. (예: `add/dijkstra`)
3. 글을 작성/수정한 뒤 PR을 엽니다.
4. 큐레이터 1명 이상의 승인을 받으면 `main`에 squash 머지됩니다.

## 글 작성 규칙

### 위치와 구조

- 모든 글은 `content/` 아래 카테고리 폴더에 둡니다.
- **한 글 = 마크다운 파일 하나** `<슬러그>.md` 입니다. (카테고리 폴더 바로 아래)
- 이미지 등 에셋이 있으면 **글과 같은 이름의 폴더** `<슬러그>/`에 둡니다. 없으면 파일 하나로 끝입니다.
- 슬러그(글 파일명·에셋 폴더명)는 소문자-하이픈(kebab-case)으로 짓습니다.

```
content/graph/shortest-path/
├── dijkstra.md            # 글 본문
└── dijkstra/              # 이미지가 있을 때만 (글과 같은 이름)
    └── example.png
```

글에서 이미지는 `![설명](dijkstra/example.png)`처럼 **글과 같은 이름 폴더 경로**로 참조합니다.

### 메타데이터 (frontmatter)

글 맨 위에 YAML frontmatter로 다음 두 항목을 둡니다.

```yaml
---
id: dijkstra          # 글의 고유 식별자 (보통 파일명과 같게, 이후 불변)
title: 다익스트라 (Dijkstra's Algorithm)   # 글 제목 (알고리즘명은 영문 병기)
---
```

- `id`는 보통 **파일명(슬러그)과 같게** 짓습니다. 다만 한 번 정하면 **불변**입니다 —
  글을 다른 카테고리로 옮기거나 파일명을 바꿔도 `id`는 그대로 둡니다.
  본 서비스(SolveMeUp)가 이 값으로 글을 참조하기 때문에, `id`가 바뀌면 연결이 끊깁니다.
- `id`는 저장소 전체에서 **유일**해야 합니다.
- `title`에는 알고리즘명의 **영문을 병기**합니다. (예: `이진 거듭제곱 (Binary Exponentiation)`)
- 태그·난이도 등 다른 필드는 추후 도입될 수 있습니다. 지금은 `id`·`title` 두 개만 둡니다.

### 카테고리

카테고리(분류 체계)는 관리자가 관리합니다. 아래 분류에 맞는 위치에 글을 추가해 주세요.
(해당 폴더가 아직 없으면 글과 함께 만들면 됩니다.)

```
content/
├── data-structure/
├── sorting/
├── graph/
│   ├── tree/
│   ├── shortest-path/
│   ├── mst/
│   ├── scc/
│   └── network-flow/
├── dp/
│   ├── knapsack/
│   ├── lcs/
│   ├── lis/
│   └── optimization/
├── string/
├── math/
│   ├── number-theory/
│   ├── combinatorics/
│   ├── linear-algebra/
│   └── polynomial/
├── geometry/
├── technique/
├── greedy/
└── game-theory/
```

- **새 카테고리가 필요하면** PR에 새 폴더를 임의로 만들지 말고, 먼저 **Issue로 제안**해 주세요.
  큐레이터 합의 후 위 목록에 추가됩니다.

### 사이드바(내비게이션)

사이드바는 폴더 구조에서 **자동 생성**됩니다. 글을 알맞은 카테고리에 추가하면 글 제목(frontmatter `title`)으로
사이드바에 자동으로 나타납니다 — **별도 등록 작업이 없습니다.** (설정 파일을 손댈 필요가 없습니다.)

- 카테고리의 한글 라벨은 각 카테고리 폴더의 `.nav.yml`에 `title:`로 둡니다. (관리자 영역)
  글이 처음 생기는 카테고리는 이 `.nav.yml`이 아직 없을 수 있는데(없으면 사이드바에
  영문 폴더명이 그대로 노출됩니다), 이때는 큐레이터가 PR에서 한글 라벨을 함께 추가합니다.
- 글 순서는 기본 **자동정렬**(파일명)입니다. 특정 순서를 고정하고 싶을 때만 그 카테고리의 `.nav.yml`에
  `nav:`로 글을 나열합니다(적은 글은 그 순서, 나머지는 자동으로 뒤). 이 역시 관리자가 카테고리 단위로 관리합니다.

### 내용

- **형식은 자유입니다.** 정해진 글 템플릿을 강제하지 않습니다 —
  알고리즘마다 자연스러운 설명 흐름이 다르기 때문입니다.
- 개념 설명과 **수도코드**를 기본으로 권장합니다.
- 언어별 구현(Java/Python/C++ 등)은 **환영하지만 필수는 아닙니다.** 한 언어만 있어도 됩니다.
- 본문은 글 제목을 **H1(`#`)으로 시작**합니다. (frontmatter `title`과 같은 제목으로)
- 수식은 KaTeX 문법(`$...$`, `$$...$$`)을 사용합니다.
- 코드 블록에는 언어를 명시합니다. (예: ` ```cpp `)
- 이미지는 글과 같은 이름의 폴더(`<슬러그>/`)에 두고 **상대 경로**로 참조합니다.
  (예: `dijkstra.md`에서 `![설명](dijkstra/example.png)`. `/img.png` 같은 절대 경로는
  쓰지 않습니다 — 사이트가 하위 경로(`/archive`)에 배포되어 깨집니다.)
- **자기 말로 서술**해 주세요. 번역·복붙이 아닌 독창적 설명을 원칙으로 하며,
  참고한 자료가 있으면 출처를 남깁니다.

## 스타일

- 한 PR은 하나의 주제에 집중합니다. (글 1개 또는 명확한 묶음 단위)
- 깨진 링크·잘못된 마크다운 문법이 없는지 확인하고 올립니다.
- 올리기 전 로컬에서 `python3 .github/scripts/validate_structure.py`로 글 구조를 미리 확인할 수 있습니다.
- 렌더 결과(수식·이미지·강조)를 미리 보려면 로컬에서 사이트를 띄웁니다:
  `pip install -r requirements.txt` 후 `mkdocs serve` → 브라우저에서 `http://127.0.0.1:8000/archive/` 를 엽니다.
- PR을 열면 글 폴더 구조와 **내부 링크/파일 경로**가 자동으로 검증됩니다. 통과해야 머지됩니다.
  (외부 URL은 검사하지 않습니다 — 시간이 지나면 자연히 죽기 때문입니다.)

질문이나 제안은 Issue로 남겨 주세요.
